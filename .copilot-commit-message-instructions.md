# Conventional Commits Guide

## Format

```
<type>(<scope>): <subject>
```

## Types

| Type       | Use When                                 |
| ---------- | ---------------------------------------- |
| `feat`     | Adding new feature                       |
| `fix`      | Fixing a bug                             |
| `docs`     | Documentation changes only               |
| `style`    | Code formatting (no logic change)        |
| `refactor` | Code restructuring (no behaviour change) |
| `perf`     | Performance improvements                 |
| `test`     | Adding or updating tests                 |
| `chore`    | Maintenance tasks, configs, dependencies |
| `build`    | Build system or external dependencies    |
| `ci`       | CI/CD configuration changes              |

## Examples

```bash
# Features
feat: add user registration
feat(auth): implement password reset

# Fixes
fix: resolve null pointer exception
fix(api): handle missing request parameters

# Simple tasks
chore: update .gitignore
chore: remove unused imports
docs: update README installation steps

# Breaking changes
feat!: redesign authentication API
BREAKING CHANGE: requires OAuth2 tokens
```

## Rules

1. Use lowercase for type and subject
2. No period at end of subject
3. Subject max 50 characters
4. Use imperative mood ("add" not "added")
5. Scope is optional but recommended
6. Add `!` after type for breaking changes
7. Format supports only a single line

## Quick Reference

**Does it change what users see or how code behaves?**

- Yes → `feat` or `fix`
- No → probably `chore` or `refactor`

**Just documentation?** → `docs`

**Just formatting?** → `style`
